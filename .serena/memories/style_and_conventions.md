# Style and conventions

## Python style
- Python code uses 4-space indentation and mostly single-quoted strings.
- Type hints are used inconsistently: some newer code annotates functions and local constants (`Path`, `list[str]`, return types), while older scripts remain untyped.
- Newer scripts often include module docstrings and concise function docstrings; older scripts may have none.
- Argument parsing is done with `argparse` in newer entrypoints.
- Path handling in newer scripts prefers `pathlib.Path`; older scripts may still pass string paths.
- `subprocess.run(..., check=True)` is the normal pattern for OWLToPDDL and Fast Downward invocations.
- Output artifacts are written into timestamped folders under `results/` or scenario-specific result directories.

## Experiment-specific conventions
- Preserve the timing rule: do not treat planner-only timings as total runtime.
- Preserve the fast serialization rule in `map_generator.py`: `robot_at` and `corridor` should keep `default_initial_value=False`, and `write_problem_fast()` is the preferred writer for current experiment problems.
- For grid experiments, connected maps are generated by construction rather than by retrying random disconnected samples.
- `scripts/runner.py` is a reusable base class module; prefer scenario entrypoints for actual runs.

## C++ / package metadata
- The ROS2 C++ package skeleton uses standard ament formatting, C++17, and warning flags (`-Wall -Wextra -Wpedantic`), but it is peripheral to the main experiment workflow.

## What not to assume
- There is no repo-configured `ruff`, `black`, `flake8`, `mypy`, or `pytest` setup visible in this repository.
- The GitHub workflow builds/pushes Docker images only; it is not a source of formatting or unit-test conventions.
